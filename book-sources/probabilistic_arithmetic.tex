\section{Probabilistic Arithmetic}
\label{sec:comp-with-rand}



When the stochastic system is sufficiently simple that its behavior can be characterized in terms of clean mathematical functions, we can obtain insight into the probability mass functions of interest by direct algorithmic computations rather than by using simulation.
There are two main advantages for this approach.
The first is that the direct computations provides us with some insight into how many runs of a simulation we need before obtaining statistically (somewhat) sound data.
If we simulate for a too short amount of time, we obtain too few sample paths to get a feeling for the statistical properties of the process we are interested in.
However, we also do not want to waste way too many computer cycles (and time) just to be sure we have not missed anything.\sidenote{As with all things in life, we need to find the right balance.}
The second reason is that building code to deal with rvs deepens our understanding of probability and numerical analysis.
This will help us prevent making subtle errors; we will point out a few in the main text and in the exercises.
We start with an example and then present the code that deals with all the computations.

\newthought{We can determine} the waiting time of the~$n$th job in a single-server queue by means of the recursion $W_{n} = [W_{n-1} S_{n-1}-X_{n}]^{+}$.
That is, for given random deviates $\{X_{i}\}$ and $\{S_{i}\}$ we can compute the \emph{number} $W_{n}$.
If want to get information about the \emph{distribution} of $W_{n}$, we need to do some more work.
One is to repeat the recursions many different times with different seeds and make a histogram of the observed values of $W_{n}$.
There is however, another way that is directly based on arithmetic with the pmfs of~$X$ and~$S$.
Let us show how to we can do this.~\cref{fig:waitingdistribution} shows the result of the code below.

Another interesting question is what would happen if we would compute the departure times with the following code, which is the implementation of the recursion $D_{n} = \max\{D_{n-1}, A_{n}\} + S_{n}$.
\begin{python}
A = rv.RV({0: 1})
D = rv.RV({0: 1})
for i in range(1, horizon):
    A += X
    D = rv.compose_function(max, D, A) + S
\end{python}
\noindent In fact, this is wrong for a subtle reason.
As will become clear from the implementation, the code in \pyth/random_variable.py/ assumes that the rvs are \emph{independent}!
However, $D_{i}$ and $A_{i-1}$ are dependent.\sidenote{Dependent is the same as not independent.}
To see this, observe from~\cref{fig:waitingdistribution} that $\P{D_{30} \leq 150} > 0$, but $\P{D_{30} \leq 150| A_{30} > 151} = 0$ while still $\P{A_{30} > 151} > 0$.

But why then does it work in the code that implements the recursion $W_i = [W_{i-1} + S_{i-1} - X_i]^{+}$?
This is because $W_{i-1}$ \emph{is} independent of $S_{i-1}$ and $X_{i}$ so that we can compute $W_{i-1}+ S_{i-1} - X_{i}$ without problems.
Keep in mind that stochastic dependence is  tricky and can be easily overlooked.
To prevent making such mistakes (when it is important) it helps to check the results against simulation.
If the results do not agree, there is an error (which may be simple or hard to find).

And this brings us to another conceptual mistake in the code that computes (but in the wrong way as we will see)
the distribution of the deparure times with the statement \pyth/D = A + W + S/.
Let us check~\cref{fig:waitingdistribution} carefully.
The ecdf (empirical cdf) obtained by simulution starts to increases later than the cdf of $D$ but increases faster.
The variance of the  simulated departure times must therefore be smaller.
Increasing the number of samples does not repair this `problem'.
So what is the matter here?
In fact, the arrival time of job $30$ and its waiting time are dependent, and \cref{thr:1} can help to understand why.
Suppose that we know that the random sample is such that $N(t) = 30$ and $t \gg S_{1} + S_2 + \cdots S_{30}$.
\cref{thr:1} tells us that the arrival times are uniformly distributed, hence in this specific case are probably be spaced out quite a bit.
Therefore, the probability must be small that an arrival occurs while another job is in service.
Similarly, suppose that $t< S_{1}$, but $N(t) = 30$.
Then all jobs, except the first, wait.
In conclusion, $A_{30}$ and $W_{30}$ are dependent, hence $D_{30}$ cannot be computed as simply as the code suggests.
In fact, its computation requires a careful build up of the dependency structure of the rvs involved.
A nice python package that implements dependence in a correct way is \pyth/lea/.

The \pyth/RV/ class, which we build in the coming code blocks, stores the pmf and support of a rv, and offers many convenience methods such as \pyth/cdf/.
We specify the support and pmf of a rv as a dictionary;  for instance, \mintinline{python}{{0: 1 / 3, 1: 2 /3 }} represents a biased coin with support $\{0, 1\}$ and pmf $p_{0} = 1/3$ and $p_{1} = 2/3$.  \pyth/RV/ use the pmf,  support and cdf as private members.
\inputminted[firstline=22, lastline=33]{python}{../code/random_variable.py} % start of rv


Dealing with the pmf as a dictionary requires considerable care.
An example helps to understand the problem.
I suppose you agree that $1/3 = 1-2/3$, so let's see what happens if we run the next code.
\begin{pyconsole}[rvs]
from collections import defaultdict

res = defaultdict(float)
res[1/3] += 1
res[1 - 2 / 3] += 1
res
\end{pyconsole}
\noindent The dictionary \pyth/res/ suddenly contains two elements rather than just one.
Inspecting the last digit of the keys, we see that in one number the last digit is a~$7$ and in the other it is a~$3$.
So, for a \emph{computer} $1-2/3 \neq 1/3$.
This is not something specific to python; the problem lies in the fact that general numbers in~$\R$ cannot be represented as floating point numbers.

To get around this, we round the elements of the support to a fixed precision so that we can use fractions instead of floats.
This works well as demonstrated by this code.
\begin{pyconsole}[rvs]
from fractions import Fraction

a = Fraction(1 - 2 / 3).limit_denominator(1000)
b = Fraction(1 / 3).limit_denominator(1000)
a, b, a == b
\end{pyconsole}

Once you understand this point, the next code is easy.

From the pmf $\{p_{k}\}$ we want to compute the cdf, the survivor function, mean, variance and standard deviation.
All this is easy once we realize that all this can be computed as $\E{f(X)}$ for suitable chosen functions~$f$.
For instance, by taking $f(x) = \1{x\leq y}$, the cdf $F(x) = \E{f(x)} = \sum_{k} \1{k \leq x} p_{k}$.
Below is an inefficient method to compute the cdf using \pyth\E/:\sidenote{Check the web on \pyth/lambda/.}: \pyth/lambda x: x <= y/.
\begin{python}
def cdf_inefficient(self, x: float) -> float:
    return self.E(lambda y: y <= x)
\end{python}

Additionally, we like to able to use our \pyth/RV/ class to generate random deviates that are distributed according to the cdf~$F$.
Recall that if $U\sim \Unif{[0,1]}$, then $F^{-1}(U)$ has the required distribution, because $\P{F^{-1}(U) \leq x} = \P{U \leq F(x)} = F(x)$, where the last equality follows from the fact that~$U$ is uniform on $[0,1]$.
Binary sort is useful and fast to compute the inverse $F^{-1}(U)$.
